# 要件定義書

## 1. 文書の目的

対象は、身内の LINE グループ内で利用する小規模なアンケート運用を前提とする。

## 2. 背景

- LINE グループ内で使うアンケートフォームを作りたい
- 不特定多数向けではなく、回答者は事前に把握できる
- 初回の本人確認や入力の手間をできるだけ減らしたい
- LINE 表示名を使った初回名寄せをあえて採用したい
- 将来的に分岐付きアンケートが必要になる可能性がある
- 最低限、CSV 出力と回答完了メールは欲しい

## 3. 目的

1. LIFF を使って LINE ログイン付きのアンケート画面を提供する
2. LINE 表示名を使って初回の回答者名寄せを行う
3. 回答者の本名・メールアドレスを自動入力できるようにする
4. アンケート項目を柔軟に定義し、回答を保存できるようにする
5. CSV 出力と回答完了メール送信を行えるようにする
6. 将来の分岐アンケートに備えて、DB 構造を拡張しやすくしておく

## 4. 対象ユーザー

- 回答者
  - LINE グループ内のメンバー
  - 事前に存在が把握できる少人数の利用者
- 管理者
  - アンケートを作成・配布・集計する担当者
  - 回答者マスターを管理する担当者

## 5. 利用前提

- 利用範囲は限定された LINE グループ内とする
- 回答者の LINE 表示名は、初回名寄せキーとして使用してよい
- 表示名の重複は事前に確認済みとする
- 回答者の本名・メールアドレスは、管理者側で事前に把握している
- 回答者の同意なく個人情報を保存しない運用にはしないため、利用目的の明示は専用ページを設ける

## 6. スコープ

### 6.1 MVP に含める

- LIFF からのログイン
- LINE 表示名の取得
- 表示名による回答者マスターの初回名寄せ
- 本名・メールアドレスの自動入力
- アンケート作成
- 設問の基本タイプ対応
- 回答保存
- 回答完了メール送信
- CSV 出力
- 利用目的・同意文の表示

### 6.2 MVP に含めない

- スプレッドシートへの自動連携
- 高度な集計・分析ダッシュボード
- 外部公開向けの大規模認証
- 不特定多数向けの一般公開フォーム運用
- 複雑なワークフロー管理

## 7. 機能要件

### 7.1 LINE 認証・識別

- LIFF から LINE ログインできること
- LINE 表示名を取得できること
- LINE userId を保存し、以後の識別に使用できること
- 以後は userId を正として扱えること

### 7.2 回答者マスター管理

- 管理者は回答者マスターを登録・更新できること
- 回答者マスターには少なくとも次の情報を保持できること
  - LINE 表示名
  - 本名
  - 敬称
  - メールアドレス
  - 備考
- 初回アクセス時に、取得した LINE 表示名で回答者マスターを検索できること
- 一致した場合は本名・メールアドレスを自動入力できること
- 一致しない場合は、管理者または利用者が補完できること

### 7.3 アンケート作成

- 管理者はアンケートを作成できること
- アンケートには次の情報を設定できること
  - タイトル
  - 説明
  - 公開状態
  - 設問

### 7.4 分岐対応

- 将来の分岐設問を考慮し、データモデルは大きく変えずに拡張できること
- 設問単位で条件分岐を表現できること
- 分岐条件は設問定義の変更だけで追加できること
- 回答データは分岐の有無に関わらず保存できること

### 7.5 回答画面

- 回答者は LIFF 内でアンケートを回答できること
- 初回名寄せ結果に基づき、本名・メールアドレスを事前表示できること
- 回答内容を送信前に確認できること
- 送信後は完了画面を表示できること

### 7.6 回答保存

- 回答送信時に回答内容を保存できること
- 回答ごとに回答日時を保存できること
- 回答者とアンケートを関連付けて保存できること
- 設問ごとの回答値を保持できること

### 7.7 CSV 出力

- 管理者は回答データを CSV で出力できること
- CSV には少なくとも以下を含めること
  - 回答日時
  - LINE 表示名
  - 本名
  - メールアドレス
  - 各設問の回答
- スプレッドシートへそのまま取り込める形式で出力できること

### 7.8 回答完了メール

- 回答送信完了時にメール送信できること
- 宛先は本名に紐づくメールアドレスを使用すること
- メール本文には少なくとも回答完了の事実が含まれること

### 7.9 同意表示

- プライバシーポリシーページへのリンクを張ること
- 保存する情報の種類を明示すること
- 管理者が誰か分かるようにすること
- 回答データの扱いを簡潔に示すこと

## 8. 非機能要件

### 8.1 拡張性

- 設問追加や分岐追加で DB の大改修が必要にならないこと
- 回答形式の追加がしやすいこと
- 将来の分析機能追加に耐えられること

### 8.2 保守性

- マスター、アンケート定義、回答データを分離して管理できること
- CSV 出力やメール送信の処理を独立して変更しやすいこと

### 8.3 セキュリティ・プライバシー

- 個人情報を扱うため、アクセス制御を設けること
- 管理者以外が回答一覧や CSV に直接アクセスできないこと
- LINE userId は内部識別子として扱い、画面上での表示は必要最小限とすること

### 8.4 可用性

- 小規模利用を前提に、シンプルで壊れにくい構成とすること

## 9. 成功条件

- 回答者が LINE 経由で迷わず回答できる
- 初回入力の手間を表示名ベースの名寄せで削減できる
- 管理者が CSV を使って集計できる
- 将来的に分岐アンケートへ拡張しても、既存回答を壊さない

## 10. 既知の制約

- LINE 表示名は名寄せキーとして弱いが、身内グループ前提のため許容する
- LINE グループのメンバー一覧の自動取得は前提にしない
- 回答者の本人確定は userId を最終基準とする
